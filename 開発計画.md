## 0. 基本方針

- **CLI で完成させてから LINE に載せる。** P1 までで「自然言語→保存→確認文」がローカルで完結する状態を作る。入口（CLI→LINE）は差し替え可能に保つ。
- **テスト駆動で進める。** DB・時刻正規化・ツール選択は壊れやすい中核。先にテストを書いてから実装する（CLAUDE.md §4）。
- **LLM 依存部分は最後まで薄く保つ。** ツール定義（JSONスキーマ）と executor はモデル非依存。LiteLLM の `model` 文字列だけで local⇄Claude を切替（NFR-3）。
- **過剰設計しない。** ルータ・マルチエージェント・複数子対応は初期スコープ外。`child_id` のカラムだけ用意し、当面は1人固定。

## 3. 実行計画（フェーズ別）

各タスクは Design Doc §11 の ID に対応。**依存** を満たした順に着手。`★` はマイルストーン到達点。

### P0: Setup — 土台を通す
| ID | 着手内容 | 完了条件 |
|---|---|---|
| T0.1 | `uv init`、ディレクトリ構成、LiteLLM/pytest 依存追加 | `uv run python -m kotolog` で hello が動く |
| T0.2 | `config.py`: `.env` から model名/APIキー/DB URL を読込 | テストで設定値が参照できる |
| T0.3 | Ollama 起動、LiteLLM 経由で1応答 | `model=ollama/...` で応答が返る |

> ⚠️ T0.3 は Ollama のローカルインストールが前提（環境依存）。未導入なら `ollama pull qwen2.5` を先に手動実行。**ここは私が代行できないので、詰まったら教えてください。**

### P1: Core（CLI）— 最重要・ここが本体 ★
| ID | 着手内容 | 完了条件 |
|---|---|---|
| T1.1 | DB層: スキーマ・接続・CRUD | `test_crud.py` 緑 |
| T1.2 | 時刻正規化: 「さっき/3時/お昼」→JST絶対時刻 | `test_timeparse.py` 緑 |
| T1.3 | ツール3関数のJSONスキーマ + executor | `test_executor.py` 緑（直接呼びでDBが変わる） |
| T1.4 | LiteLLM ラッパ（model切替） | 設定だけで local/Claude 切替できる |
| T1.5 | エージェント・ループ（tool-use + 確認サマリ + 聞き返し） | 自由文→保存→確認文が返る |
| T1.6 | 対話CLI | CLIで記録・集計・修正ができる ★**P1完了** |
| T1.7 | ツール選択の評価テスト | 代表入力が期待ツール/引数を呼ぶ |

**P1 の進行順（依存解決）:** `T1.1 ∥ T1.2` → `T1.3` → `T1.4`（T0.3後）→ `T1.5` → `T1.6` → `T1.7`
（T1.1 と T1.2 は独立なので並行可。T1.4 は T0.3 完了後ならいつでも）

### P1.5: MVP+ — 毎日使える集計体験へ ★

出典: [docs/competitive-analysis.md](docs/competitive-analysis.md)（既存アプリの機能比較）。

**設計方針（重要）:** ツールは3つのまま増やさない（選択精度を守る）。件数・合計・経過時間
などの**計算は executor が確定値で返し、LLM は文章化だけ**にする（時刻のサーバ側正規化と
同じ思想）。下記はすべて既存ツールへの「引数追加＋戻り値強化」で収まる。

| ID | 着手内容 | 依存 | 完了条件 |
|---|---|---|---|
| T1.8 | **query_records の集計化**: 戻り値に種別別(`by_type`)・sub_type別(`by_sub_type`)の件数/合計を追加。任意 `sub_type` フィルタ引数を `definitions` と executor に追加 | T1.3 | executor 単体テスト緑（複数記録で by_type/by_sub_type の件数・合計が正しい） |
| T1.9 | **sub_type の正規化**: 保存時に同義語を正規形へ寄せる（母乳/ミルク/搾母乳・うんち/おしっこ 等）。集計の一貫性を担保 | T1.3 | 正規化ユーティリティの単体テスト緑（表記ゆれが正規形に集約） |
| T1.10 | **1日/週まとめ**: 「今日のまとめ」「今週どうだった?」で query を1回呼び、全種別集計を要約文に。プロンプト＋agent側の調整 | T1.8 | 決定論E2Eで、まとめ要求時に集計が1回のquery結果から文章化される |
| T1.11 | **前回いつ／何時間前**: 直近1件＋経過時間(分/時間)を返す `latest` 問い合わせを追加（period に `latest` を追加し executor で経過を確定計算） | T1.8 | executor 単体テスト緑（直近＋経過を返す）。「前回の授乳いつ?」で latest 問い合わせが選ばれる |

**P1.5 の進行順:** `T1.9 ∥ T1.8` → `T1.10` / `T1.11`
（T1.8/T1.9/T1.11 の executor ロジックは LLM 不要でテスト先行可。T1.10 と「期待ツールが選ばれるか」の確認は実モデル／eval が必要）

> 補足: 記録種類の拡張（体重/身長・くすり/病院・症状）は P1.5 完了後に enum 追加で漸進対応。
> 区間タイマー・共有・一括入力は思想/構造の都合で保留（competitive-analysis.md §3-4 参照）。

## 5. 着手順序（推奨）

```
今ここ → T0.1（雛形）── 私がすぐ着手可能
        → T0.2（config）
        → T1.1 + T1.2（並行・テスト先行）  ← LLM不要、ここまで一気に進められる
        → T1.3（ツール）
        → T0.3 + T1.4（Ollama疎通・要ローカル環境） ← あなたの環境作業が必要
        → T1.5 → T1.6（CLI完成 = M1）
```

**ポイント:** T0.3/T1.4 以外（T0.1, T0.2, T1.1, T1.2, T1.3）は LLM もネットワークも不要で、私が連続して実装・テストまで完了できます。Ollama 疎通だけがあなた側の環境準備に依存します。
