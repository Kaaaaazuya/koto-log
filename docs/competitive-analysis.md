## 1. 既存タップ型アプリの機能傾向

| カテゴリ | 一般的な機能 |
|---|---|
| 記録項目 | 母乳/ミルク/搾母乳・離乳食/おやつ・うんち/おしっこ・睡眠・体温・身長/体重・お風呂・さんぽ・症状(せき/発疹/嘔吐)・けが・くすり・病院・自由メモ・写真日記 |
| 入力効率 | ワンタップ記録・授乳タイマー(左右計測)・ウィジェット・音声アシスタント・片手操作 |
| 可視化 | 自動グラフ・タイムバー(時系列)・週まとめ・成長曲線(修正月齢) |
| 共有 | 家族間のリアルタイム共有・前回記録の確認(夜間交代) |
| 通知 | 授乳時間などのリマインダー |
| 出力 | PDF/製本・記録検索 |
| その他 | 複数児対応・項目の表示/非表示・ダークモード・予防接種スケジュール |

### タップ型UIの構造的な弱点（差別化の手がかり）
- **定型の選択肢に縛られる** — リストにない品目や例外的な記録（自費の項目など）を
  自由に残しにくい。
- **“状況”は自由メモ止まり** — 「30分で起きてぐずってた」のような文脈は構造化されず、
  集計対象にならない。
- 項目数が多く、初期の学習コストがある。

## 2. koto-log に足りない機能（優先度付き）

koto-log の `records` スキーマ（`type / sub_type / amount / unit / started_at / ended_at / note`）は
既に汎用的で、多くの項目は **enum 追加だけ**で吸収できる。これが効く。

| 優先 | 機能 | koto-log での実装 |
|---|---|---|
| **高(MVP+)** | 母乳/ミルク/搾母乳の区別、うんち/おしっこの区別 | `sub_type` 値を増やすだけ。集計の実用性に直結 |
| **高** | 区間記録の開始/終了フロー（睡眠・授乳タイマー相当） | 「ねんね開始」→`started_at`保存、「起きた」→直近を`ended_at`で締める。`update_or_delete` を拡張 |
| **高** | 1日サマリ／週まとめ | LLM の得意領域。`query_records` の集計を文章化 |
| **高** | 「前回の授乳いつ？」「前回から何時間？」 | `query` に直近1件＋経過時間を返すケースを追加 |
| 中 | 体重/身長(成長)、くすり/病院/症状(せき/発疹/嘔吐) | `type` 追加。`amount+unit`(kg/cm)で対応。受診時の説明に効く |
| 中 | 複数記録の一括入力（「3時ミルク120、5時うんち」） | ループで複数 tool-call。プロンプト調整 |
| 中(P2/P3) | 共有（家族で同じ子） | 複数 `line_user_id` → 同一 `child_id` のマッピング。設計拡張 |
| 低(P4) | グラフ画像返信 / リマインダー | 既に計画済（T4.3 / T4.2） |

## 3. koto-log ならではの差別化（ここで勝つ）

既存タップ型アプリの**機能の幅を真似するのは負け筋**。別の軸で勝つ：

1. **“状況”をそのまま記録できる** — 「寝たけど30分で起きてぐずった」を自由文で保存し、
   `note` ＋構造化を両立。集計にも乗せられる。
2. **記録と相談の統合** — 「38度ある、大丈夫？」に対し、記録しつつ断定せず受診を促す
   （DesignDoc §9 のガード既設）。記録専用アプリにはできない体験。
3. **会話型の振り返り** — 「最近寝つき悪い？」のような曖昧な問いに蓄積データから答える。
4. **“例外”に強い** — 定型の選択肢に縛られず、自由入力で何でも残せる。

---

## 4. スコープ判断（非ゴール維持を推奨）

DesignDoc の非ゴールと照らし、**あえて作らない**もの：
- 予防接種の**スケジュール管理**（別アプリ級。“記録”だけは自由入力で軽く拾える）
- 成長曲線のパーセンタイル/修正月齢の専用表示（過剰設計）
- 写真日記（LINE で画像受信は可能だが保存コスト/スコープ外）

---

## 5. 次の一手（提言）

優先度「高」の4つ — **sub_type 拡充 / 区間記録フロー / 1日・週まとめ / 前回経過** が、
最小工数で「実際に毎日使える」に最も効く。いずれも既存スキーマ＋プロンプト/executor の
小改修で収まるため、開発計画の **P1.5（MVP+）** として切り出す。
