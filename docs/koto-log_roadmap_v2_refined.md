# koto-log ロードマップ ブラッシュアップ（v2）

*作成日: 2026年6月28日 / 前提: **自分・家族で使う個人ツール** / 参加ロール: PdM・テックリード・PO*

> v1（競合調査 & ロードマップ）を、PdM・テックリード・PO の3者レビューで磨き直したもの。
> 最大の前提変更は「**これは B2C プロダクトではなく、自分と家族が使う個人ツール**」という点。
> この前提により、v1 に含めていたマルチテナント・自治体連携・課金・ML 付加価値などを意図的に外し、
> **「技術的負債の解消」と「不足している基本機能の追従」をどう両立させるか**に論点を絞った。

---

## 1. 3者の基本スタンス

**PdM（基本機能の追従は MUST）**
「自分の家族が、koto-log だけで完結できる状態がゴール。今は記録項目が4種しかなく、成長曲線も予防接種リマインドもない。これだと結局ぴよログを併用することになり、koto-log を使う意味が薄れる。**“他アプリに浮気しなくて済む基本機能”の追従は譲れない。**」

**テックリード（技術的負債の解消を重視）**
「機能を足す前に土台を直したい。特に `KOTOLOG_DEFAULT_CHILD=baby` の単一子ども前提と単一ユーザー前提は、項目や子どもが増えた瞬間に効いてくる**構造的負債**。スキーママイグレーションの仕組みもなく、agent ループにはフォールバック解析が残っている。**負債の上に機能を積むと、後の変更コストが指数的に増える。** ただし個人ツールなので、エンタープライズ級の作り込みは過剰だ。」

**PO（スコープと優先順位の番人）**
「2人とも正しい。だが個人ツールである以上、**“やらないこと”を決めるのが自分の仕事。** マルチテナント、課金、自治体連携、製本、泣き声診断のような ML 機能は今回スコープ外。そのうえで、**各フェーズに『負債を1つ返す』と『基本機能を1つ足す』を同居させて**、どちらかが先送りされ続ける事態を防ぎたい。」

---

## 2. 主要論点と合意

### 論点A: データモデル刷新は「どこまで」やるか

- **PdM**: 夫婦で共有できないと家族ツールとして不完全。共有は MUST。
- **テックリード**: 共有とマルチ子どもは同じ根（`user`–`child` の関係）。ここは先に直したい。
- **PO**: ただし**個人ツールなので“1家族・数人・数人の子ども”で十分**。招待リンクの大規模基盤やアカウント管理、権限ロールは作らない。

**合意**: データモデルは刷新するが、**スコープは「1世帯・LINEユーザー数名・子ども複数」に限定**。共有は admin 画面での LINE ユーザー登録（数人）で実現し、本格的な招待/認証基盤は作らない。`KOTOLOG_DEFAULT_CHILD` 前提は撤廃し、会話内で対象児を解決（未指定はアプリ側で既定児にフォールバック、LLM に推測させない原則は維持）。

### 論点B: 記録項目はどこまで増やすか

- **PdM**: ぴよログ並みの網羅（離乳食・薬・病院・お風呂・身長体重…）が欲しい。
- **テックリード**: 項目を増やすほど `subtype` 正規化と eval が膨らむ。むやみに全部入れると精度が落ちて保守が重くなる。
- **PO**: 個人ツールなら「**自分の家族が実際に使う項目**」だけでいい。全網羅は不要。

**合意**: 「**自家で使う項目**」を選定して段階追加（例: 離乳食・薬・病院・お風呂・搾乳・授乳の左右）。**項目追加と同時に eval ケースを必ず足す**ことを運用ルール化（テックリードの要求＝回帰防止）。網羅性の競争はしない。

### 論点C: 技術的負債をどこまで返すか（個人ツールの適正水準）

- **テックリード**: マイグレーションの仕組み、フォールバック解析の整理、コストの可観測は入れたい。
- **PO**: Langfuse 移行のような重い可観測は自家用には過剰。**「壊れにくい・直しやすい」最小限**に留める。
- **PdM**: 負債解消それ自体はユーザー（家族）に見えないので、**必ず機能とセットで**進めてほしい。

**合意**: 返す負債を3点に絞る — ①スキーママイグレーションの仕組み導入（今後の項目追加に効く）、②agent ループのフォールバック解析の整理、③**トークンコストの軽量可観測**（個人の API 課金に直結するため。Langfuse 移行は任意）。各負債は単独フェーズにせず、関連機能と同じフェーズに同居させる。

### 論点D: スコープ外にするもの（PO の決定）

個人・家族用ツールという前提から、次は**今回やらない**と明確化した。これにより v1 から軽量化される。

| v1 にあった項目 | 今回の扱い | 理由 |
| --- | --- | --- |
| マルチテナント / 本格的な権限基盤 | **スコープ外** | 1家族で使うので不要。過剰設計 |
| 自治体・子育て支援制度連携 | **スコープ外** | B2C/行政向けの世界。個人ツールに不要 |
| 課金 / プレミアムプラン | **スコープ外** | 売る相手がいない |
| 泣き声診断などの ML 付加価値 | **スコープ外** | 開発・運用コストに見合わない |
| 製本・本格エクスポート導線 | **後回し（任意）** | あれば嬉しい程度。優先度低 |
| Langfuse 等の本格可観測 | **任意（軽量ログで代替）** | 自家用には軽量ログで十分 |

## 3. 改訂版ロードマップ（v2）

PO の原則「**各フェーズに『負債を返す』×『基本機能を足す』を同居**」で再構成。既存の P1〜P8 を継承し P9 以降とする。

### P9. 基盤の最小刷新 ＋ 子ども複数・夫婦共有（軽量）

- **基本機能（PdM）**: 子どもを複数登録、夫婦など数名の LINE ユーザーが同じ子を記録・参照。会話で対象児を切替（「上の子のミルク…」を解釈）。
- **返す負債（テックリード）**: `KOTOLOG_DEFAULT_CHILD` 前提の撤廃、`user`–`child` データモデルへ移行、**スキーママイグレーションの仕組み導入**。
- **個人ツールの線引き（PO）**: 招待リンク基盤・認証・権限ロールは作らない。admin で数名の LINE ユーザーを登録する方式で十分。ADR 化。

### P10. 記録項目の拡充（自家で使う範囲）

- **基本機能（PdM）**: 離乳食・薬・病院・お風呂・搾乳・授乳の左右など、家族が実際に使う項目を選んで追加。自然言語入力なので UI 改修はほぼ不要。
- **返す負債（テックリード）**: `RecordType` enum・ツール定義・`subtype.py` 正規化辞書の整理。**追加項目ごとに eval ケースを並行追加**（ツール選択精度の回帰防止）。
- **線引き（PO）**: 全網羅はしない。使われない項目は足さない。

### P11. 成長記録・成長曲線

- **基本機能（PdM）**: 身長・体重の記録、厚労省/WHO 標準値と比較する成長曲線、`/dashboard` に成長タブ。親の関心が高い定番。
- **返す負債**: ダッシュボード（P4）の Chart.js 基盤を再利用し、グラフ生成ロジックを共通化。
- **線引き（PO）**: 標準値は静的データ同梱で十分。外部連携しない。

### P12. 予防接種・健診リマインド

- **基本機能（PdM）**: 生年月日から接種スケジュールを生成し、接種前・健診前に LINE Push。koto-log の**プロアクティブ通知（P5）の強みが最も活きる領域**。
- **返す負債（テックリード）**: APScheduler のジョブ拡張、スケジュール定義を設定データとして外部化（改定追従しやすく）。
- **線引き（PO）**: デジタル予診票や自治体連携には踏み込まない。あくまで自家のリマインド。

### P13. リマインダー・振り返り高度化 ＋ コスト可観測

- **基本機能（PdM）**: 授乳・おむつの間隔リマインダー（前回から N 時間）、月次サマリーの自動生成（LLM 文章＋グラフ）、写真付き日記（任意）。
- **返す負債（テックリード）**: **トークンコストの軽量可観測**（operation 別の JSON ログ＝P7 の延長で十分）、プロンプト/モデル選択のコスト最適化。個人の API 課金を見える化。
- **線引き（PO）**: Langfuse 等の本格基盤は入れない。写真日記はストレージコストを見て任意判断。

### P14（Later・任意）. 保守・品質の継続改善

- eval の継続拡充とツール選択の回帰テスト自動化、依存ライブラリ更新、agent ループの**フォールバック解析の整理/削減**（精度が安定したら簡素化）。
- 1人保守の個人ツールとして「壊れにくく・直しやすい」状態を保つための、薄く長い活動。

## 5. v1 からの主な変更点

1. **前提を「個人・家族用ツール」に確定**し、マルチテナント/自治体連携/課金/ML 機能/製本を**スコープ外**に。ロードマップが大幅に軽量化された。
2. **P9 のデータモデル刷新を「1家族・数名・子ども複数」に縮小**。本格的な招待・認証・権限基盤を作らないと明記（テックリードの“過剰設計回避”と PO の“やらないこと決定”が合致）。
3. **各フェーズを「負債1つ × 機能1つ」の同居構成に再編**。負債解消が単独で先送りされる／機能だけが積み上がる、のどちらも防ぐ（PO の調停案）。
4. **記録項目を“全網羅”から“自家で使う範囲”へ**。項目追加と eval 追加をセットで運用ルール化（PdM の MUST とテックリードの回帰防止を両立）。
5. **可観測性を軽量路線に**。Langfuse 等は任意とし、P7 の JSON ログ延長でコストを見える化（個人の API 課金管理が主目的）。

---

## 6. 直近の着手順（合意事項）

1. **P9 のデータモデル設計を ADR から開始。** 機能を増やす前の今が最も安く移行できる、という点で3者が一致。スコープは「1家族・数名・子ども複数」に限定する。
2. **P10 を P9 直後に。** 自然言語入力ゆえ低コストで基本機能差を縮められ、ROI が最も高い（PdM 最優先）。項目追加には必ず eval を添える。
3. **P12 のリマインドは P5 基盤で早期に小さく試す。** 「先回りして助ける」koto-log のキャラクターが最も立つ領域。
4. **負債は機能に相乗りさせる。** 単独の“負債返済フェーズ”は P14 を除き作らない（個人ツールでは止まりやすいため）。
